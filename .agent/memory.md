## Stack (M1 trusted core) — researched SOTA, deliberately overriding the outline's human-popular defaults
- Runtime: `uv` src-layout single pkg `verifier` (+`py.typed`, `uv_build`), committed `uv.lock`. `requires-python = ">=3.13,<3.14"` AND committed `.python-version` = 3.13 both pin the 3.13.x line. Cap (not just a floor) for reproducibility: msgspec ships per-interpreter compiled wheels and stdlib behavior can drift across minors, so a bare `>=3.13` floor would silently admit an untested interpreter. The pin is necessary, NOT sufficient — determinism is ASSERTED by golden hex vectors (the actual canonical-form guard), and the runtime stamps full Python/msgspec/unidata versions + `canon_version` into the VCert for provenance. Both layers (host `.venv-host`, container `.venv`) honor `.python-version`.
- Service (M2): **Litestar** + **uvicorn** single-worker 127.0.0.1 (over FastAPI: msgspec-native end-to-end — response structs render + OpenAPI-document natively, no second model stack, less glue; raw `request.body()` exists in FastAPI too, so the pick = ergonomics NOT a security property). Fail-closed invariant: POST bodies reach `decode_spec` as RAW BYTES via `await request.body()` ONLY — PROBED (Litestar 2.24.0): `data: bytes` is framework-PARSED (JSON-decoded first, dup keys silently collapse, then 400 "Expected bytes, got object"); `request.body()` = byte-exact. CPU-bound work: async handler + `litestar.concurrency.sync_to_thread(fn, *args)` (ParamSpec-typed idiomatic wrapper, no direct anyio dep — M2.2-CONFIRMED). `@post` defaults 201 → explicit `status_code=200`. Body cap = built-in `request_max_body_size` (app+handler param; probed 413 at the `request.body()` read — handler entered, content-type guard already run, NOT pre-dispatch — for content-length AND chunked; 2.24's `create_test_client` rejects the kwarg → build the app, wrap `TestClient(app=…)`). Error split: verification outcomes incl. decode failure (expected model failure mode, M3 meters it) = 200 verdict envelope; ONLY transport misuse/server-config = RFC 9457 `application/problem+json`. M2.2-CONFIRMED impl: two `exception_handlers` `{HTTPException → problem+json(exc.status_code, exc.detail); Exception → generic 500, cause WITHHELD from the untrusted caller}` (the `Exception` handler catches a pipeline raise even under `raise_server_exceptions=True`, probed); content-type guard = `request.content_type[0]` (pre-lowercased media-type essence, params/charset stripped, `""` if absent) `== "application/json"` else 415. OpenAPI: `create_examples=False`; explicit `operation_id`+`summary` per route.
- M4-forward (researched at M2 plan; REVERIFY at M4 planning — Open WebUI moves fast): Open WebUI maps `operationId`→tool name and reads `summary` NOT description; wants a single flat JSON-object request body; USER-registered tool servers are called from the BROWSER (CORS for the Open WebUI origin required; HTTPS UI → tool on https or localhost) while ADMIN/global ones call from the backend (Docker `localhost` = the container). Serving HTML/SVG directly later: allowlisted Content-Type + `X-Content-Type-Options: nosniff` + `Content-Security-Policy: sandbox` as HTTP HEADERS (meta tag ineffective), never `allow-same-origin`.
- Schema: **msgspec** (NOT Pydantic) — strict/fail-closed, `forbid_unknown_fields` on every `Struct`, transform ops = tagged union (`tag_field="op"`), `Literal` enums, module-level `Decoder`, `json.schema` export golden-snapshotted.
  - **msgspec confirmed behaviors** (empirically pinned for our msgspec; this cost a full window of research — read and transcribe, don't re-derive; `schema.py` cites these by finding number): (1) `frozen`/`forbid_unknown_fields` propagate to subclasses at RUNTIME but `kw_only` does NOT; and mypy's dataclass_transform reads each class's own kwargs — so EVERY concrete struct must repeat `frozen=True` (for `mypy --strict`, else "non-frozen dataclass cannot inherit from frozen") AND `kw_only=True` (runtime kw-only, since it does not inherit). A `__struct_config__` test backstops frozen+forbid_unknown_fields but does NOT expose `kw_only`, so a separate test asserts positional construction raises `TypeError`. (2) For the transform tagged union, set `tag_field="op"` + an explicit lowercase `tag` per member (else msgspec defaults the tag to the class name and the field name to `type`). (3) Strict mode rejects float→int and bool→int but ACCEPTS a JSON float→`Decimal` — so model spec numerics as `int | str` (NOT Decimal) to reject float/bool tokens at decode; decimals travel as strings, lifted per manifest at eval; the int is further bounded to signed int64 (`Meta(ge=-(2**63), le=2**63-1)`) — out-of-range or fractional numbers travel as the bounded string. (4) Duplicate object keys silently LAST-WINS, no switch — reject via a stdlib `json.loads(raw, object_pairs_hook=…)` pre-scan after the msgspec decode (input is already well-formed JSON). (5) `msgspec.json.schema()` emits NO `$schema` (stamp the Draft-2020-12 URI ourselves), is deterministic cross-process, sorts `Literal` enums alphabetically, renders a tagged union as `anyOf`+OpenAPI `discriminator`+single-element `enum`, maps `forbid_unknown_fields`→`additionalProperties:false`. The exported golden is ADVISORY, not the gate — JSON Schema `integer` admits `1.0`/`1e3` (zero-fraction floats) that strict decode rejects, so `decode_spec` stays authoritative; pop+re-add `$schema` so it sorts last even if a future msgspec emits one. (6) `msgspec.field(name=…)` renames in decode+encode+schema and rejects the un-renamed key (used for `type`→`kind`, `as`→`output`); when an attribute is itself named `field` (`Channel.field`, `Measure.field`, …) call it as `msgspec.field(…)` via the module — the bare `field:` annotation shadows the imported `field` for mypy ("str not callable"), though it still works at runtime (annotation-only names don't bind). (7) A frozen struct holding a `list` is only SHALLOWLY immutable (`spec.transform.append(…)` still mutates) and unhashable — so model every JSON array as a bounded `tuple[T, ...]` (`Annotated[..., Meta(min_length/max_length)]`; msgspec decodes JSON arrays to tuples and enforces the length), making specs DEEPLY immutable and fully hashable. Test immutability via an attribute set raising `AttributeError`. (8) `Encoder(order="deterministic")` keeps struct field order while sorting dict/set keys, renders Decimal→string, does NO Unicode normalization (→ the spec hash).
- Evaluator: **hand-rolled pure-Python** (NOT pandas/polars) — the trusted core; total control of row/col order, nulls, Decimal-exact accumulation (Fraction Σ, one HALF_EVEN quantize).
- Hashing — all SHA-256 over **text** canonical forms, NEVER Parquet/Arrow bytes (version-coupled): table = typed-NDJSON (header `name:type`, fixed col order, total-sort rows, fixed per-type formatters, one null sentinel); spec = msgspec re-encode of the validated struct (definition-order fields, byte-faithful strings — NO Unicode normalization anywhere, floats forbidden in specs, `canon_version` tag — leaner than JCS/rfc8785, sufficient since we own the encoder); dataset = raw CSV bytes (byte-exact SOURCE identity, NOT permutation-invariant); + per-column manifest hash over raw bytes. Only the plotted-table hash is permutation-invariant (total-sort closure). NFC was REVERSED by codex review: folding collides specs that select DIFFERENT rows (eval + oracle compare filter strings verbatim by UTF-8 code-point order) and would force NFC into eval+oracle too. Golden hex vectors lock the canonical bytes against drift from ANY source (msgspec/Decimal/Python/locale); re-derive them via a scratch `compute_goldens.py` with stdout redirected to a file (Headroom compresses inline tool output).
- Oracle (dev/test only): **DuckDB** `threads=1`, columns as matching `DECIMAL(38,scale)` — CONFIRMED byte-identical to eval on all goldens + adversarial synthetics; outside its DECIMAL(38)/HUGEINT domain it raises LOUDLY at pinned sites (filter-literal magnitude; SUM-accumulator / typed-reinsert overflow), never silently diverges. Good specs bake the live `sha256(raw CSV)` → editing a CSV breaks that binding BY DESIGN.
- Bad-spec mono-causality is COUNTERFACTUAL: each fails exactly one way even if you imagine fixing that defect → choose dataset/fields so no second defect lurks. `caught_by` stays PROVISIONAL where §9 keeps the semantic layer joint (M1.4 eval + M1.5 checks).
- Good specs aggregate only unit-bearing fields with unit-PRESERVING fns; a derived column inherits `unit`/`label` through its measure to the source (§7). `count` is dimensionless → unit-EXEMPT (`label.quantitative_units_present`); no good-spec golden uses `count` — the exemption is exercised by dedicated `test_checks.py` specs, so adding one would ripple M1.4e/f goldens.

## M1 lessons (forward; unit blow-by-blow lives in git — `git log --grep "(M1[. ]"`)
Process / right-sizing (rule itself → roadmap; it binds M2+ planning):
- Overflow modes seen, with remedies: design-too-wide → split at a coverage-safe seam; re-derivation despite a complete recipe → TRANSCRIBE, don't re-derive (trust verified goldens); probing a native dep in the implementing window → isolate probes to scratch sessions and pre-bake a gate-validated recipe.
- An investigation-heavy codex-review can overflow while the fix stays small → capture the verified fix as a re-appliable patch/recipe + verdict, splitting investigation from application.
- A claim-narrowing fix must SWEEP every restatement of the phrase across src+tests+memory — one narrowed line beside stale twins reads fixed but isn't.
Test design:
- A green INTERNAL gate (lint/type/100%-branch/structural asserts) does NOT prove EXTERNAL-contract validity → test output artifacts against the real consumer (jsonschema; a full vl-convert COMPILE — now a standing gate over all good specs).
- 100% BRANCH coverage misses data-DOMAIN gaps (the prec-28 rounding bug) and cannot pin LOOKUP-TABLE entries → regression-test the real data path at hostile magnitudes; matrix-test an enumerable key-space against the spec restated in-test (non-circular oracle).
- Decimal aggregation must be context-free EXACT (`Fraction`/scaled-int), never bare `sum()`/`+` — the ambient context (prec 28) rounds intermediates → accumulation-order-dependent, breaking exactness + permutation-invariance.
- Order-sensitivity as a Hypothesis property is HOLLOW (tautological) → one concrete anchor witness; a PYTHONHASHSEED-stability guard needs TIE-HEAVY rows + several seeds else it passes vacuously; "proves" overstates a bounded sample → "samples / finds no counterexample".
- Direct-helper test fixtures must be REALIZABLE pipeline states (not phantom ones); an acceptance criterion naming a multi-step scenario needs an END-TO-END pin through the public entry, not only the algorithm helper.
- cast-not-assert: narrow `X | None` via `cast` at a gated use site — an `assert`'s never-taken branch fails the 100% gate. msgspec `Meta` constraints enforce at DECODE only (`structs.replace`/direct construction skip them) → any security gate resting on a decoded-shape invariant must re-hold at the use site (path confinement = authoritative `resolve()`+`is_relative_to`). BUT when a model string enters a path only via `Path(name).stem` or a `fullmatch` id regex, confinement is BY CONSTRUCTION → NO runtime `is_relative_to` branch (unreachable → fails warn_unreachable + 100%-branch); the authoritative form is needed only where the WHOLE model string builds the path (checks.py CSV). M2.2 dropped its roadmap-planned confinement branch on this basis (pipeline.py manifest path = `.stem`).
- Per-element uniqueness ≠ global acyclicity: a backward walk over a schema-REBUILDING pipeline keys on (name, POSITION), never name alone — else non-terminating or unsound on reused output names.
- A green parity suite ≠ a correct oracle: a probe-validated draft re-parsed source through the banned forced-DECIMAL CAST and clean goldens hid it → verify the oracle INGESTS the shared coercer's cells, and feed adversarial synthetics beyond the golden shapes.
- An equivalence differential must compare FULL messages + the cross-product of independent failure modes — a check-tag-only diff missed a magnitude/precision precedence flip.
- A test embedding a symbol name as a substring → do renames via TARGETED edits, never `replace_all`.
Refuted probes (do NOT re-investigate; evidence in git): DuckDB `executemany` does NOT route large in-domain Decimals through float64; whole-table + all-null-group aggregates agree with eval; native Decimal binds correctly across its full range EXCEPT positive-exponent literals (the oracle filter path coerces via `ingest._coerce_numeric`, exp≤0). Planning-era "one spec per render (Vega element-id counter)" concern refuted — in-process SVG byte-determinism holds (tested).
